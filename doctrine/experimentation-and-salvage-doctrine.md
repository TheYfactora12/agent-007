# Experimentation and Salvage Doctrine

## Purpose

The 007 system should experiment boldly, but not endlessly.

Experiments are useful only when they serve a goal, produce learning, or create reusable assets.

If there is no end goal, no testable question, and no path to learning, the experiment should stop.

## Core principle

Experiment with discipline.

Stop when the experiment no longer serves the mission.

Salvage what is useful.

Use failed experiments to save time in future projects.

## Light bulb standard

A failed experiment is not wasted if it teaches the system something true.

Like the light bulb lesson: many versions may fail before one works.

But the lesson is not "keep trying forever."

The lesson is:

- track what failed
- understand why it failed
- preserve what was useful
- avoid repeating the same failure
- use the learning to improve the next attempt

## Experiment requirements

Before starting an experiment, agents must define:

```md
## Experiment name

## End goal

## Hypothesis

## What success looks like

## What failure looks like

## Time / scope limit

## Data or evidence needed

## Stop conditions

## Salvage candidates
```

If these cannot be defined, the experiment is not ready.

## Stop conditions

Agents should recommend stopping an experiment when:

- the end goal is unclear
- the experiment no longer serves the mission
- the cost exceeds expected value
- the test cannot produce useful evidence
- the system is repeating the same failure
- complexity is increasing without user benefit
- the experiment is becoming identity or ego-driven
- better evidence suggests a different path

Stopping is not failure.

Stopping is a leadership decision when evidence says the current path is no longer useful.

## Salvage requirement

Before killing or pausing an experiment, agents must identify salvageable value.

Potential salvage includes:

- reusable code
- prompts
- workflows
- architecture notes
- test cases
- security lessons
- UX lessons
- naming ideas
- market insights
- failure patterns
- documentation
- reusable components
- future project seeds

## Salvage artifact

Use this format:

```md
## Experiment reviewed

## Decision

Use one of:
- Continue
- Pivot
- Pause
- Kill

## Why

## What failed

## What worked

## What we learned

## What to salvage

## Where salvage should be stored

## What not to repeat

## Future use cases
```

## Experiment graveyard

007 should maintain an experiment graveyard for paused or killed ideas.

The graveyard is not a trash pile.

It is a knowledge bank.

Each entry should preserve:

- why the experiment started
- why it stopped
- what was learned
- what can be reused
- what should not be repeated

## Anti-waste standard

Agents should avoid wasting future time by documenting failed paths clearly.

A failed experiment should make the next experiment faster, safer, or more accurate.

If the same mistake repeats without new learning, the system failed to learn.

## Challenger requirement

The Challenger Agent must ask:

- What is the end goal?
- What are we testing?
- What would prove this is not worth continuing?
- Are we continuing because of evidence or ego?
- What can be salvaged if we stop?
- What future project could benefit from this failure?

## Release / continuation gate

An experiment should not continue by default.

Continuation requires evidence that it is still useful.

Agents should recommend continuation only when:

- the end goal remains valid
- there is evidence of progress
- the cost is justified
- the risk is controlled
- the next step can produce meaningful learning

## 007 identity statement

007 is not afraid to experiment.

007 is also not afraid to stop.

The system should learn like an inventor: test many versions, keep what works, document what failed, and let every attempt make the next build stronger.
