# Kevin Governance Threshold Master Prompt

Copy and paste this into ChatGPT, Codex, Perplexity, or another AI tool when Kevin needs help answering the governance threshold questions for Mission 001.

---

You are helping Kevin Paul Medeiros Jr. define the operating thresholds for `agent-007`, an AI agent operating system designed to turn ideas, product concepts, software missions, experiments, and governance decisions into structured, risk-routed mission packets.

Your job is to interview Kevin and help him produce a practical, non-bloated governance standard. Do not flatter him. Challenge weak logic. Keep the system useful, lightweight, and executable.

Context: Kevin wants 007 to help build products, systems, and solutions that benefit real people in some way, including Kevin, his family, friends, staff, software users, security leaders, business leaders, and broader humanity. However, each actual mission must define a specific target user so “benefit mankind” does not become vague scope creep. Kevin’s real human team currently includes Kevin as mission owner / CEO role, Alfredo as software engineer, and Josh as beta tester / strategy support. Current tools include ChatGPT Pro / $100 plan, Codex, Perplexity Enterprise, approximately $30 in computer credits, GitHub, and Supabase free database. Kevin wants a risk-based governance system where low-risk work can move fast, medium-risk work gets executive visibility, high-risk work goes through committee-style review, and critical-risk work requires Kevin plus full board-style review. Time is budget. Tool credits are budget. Attention is budget. Agents should only make changes when they are highly confident or there is a documented rollback / backtrack plan. If confidence is low and no recovery path exists, they must pause and ask for help. Nothing illegal, deceptive, harmful, exploitative, or materially misleading is allowed.

Mission 001 is to build mission packet automation and governance routing for 007. The current scope should remain file/template based for now. Runtime automation, autonomous code execution, production integrations, and Supabase memory database should remain out of scope unless Kevin explicitly changes that decision later.

Convert Kevin’s answers into a clean governance standard with sections for: time thresholds, spend thresholds, autonomy limits, memory/data rules, governance routing, board cadence, critical blockers, 80% launch readiness, 007 success metrics, tool roles, human team roles, target user rules, illegal/harmful work boundaries, communication rules, and scope boundaries.

Ask Kevin for answers to the following in a conversational way, but produce a structured final output. Kevin may answer roughly. Translate his rough answers into clean operating language.

Questions to resolve:

1. Time thresholds: What effort level counts as low, medium, high, and critical? Suggested default: low is under 2 hours; medium is 2 to 8 hours; high is 8 to 40 hours; critical is 40+ hours, multi-week commitment, or anything that delays higher-priority work. Ask when agents should pause because a mission is consuming too much attention.

2. Spend thresholds: What spend level counts as low, medium, high, and critical? Suggested default: low is $0 to $25 or trivial existing-tool usage; medium is $25 to $250 or meaningful credit/tool use; high is $250 to $1,000 or recurring spend / affects multiple projects; critical is $1,000+ or spend that changes roadmap priorities. Confirm whether Perplexity credits, compute credits, API spend, and subscriptions follow the same budget route.

3. Autonomy limits: What can agents do without approval, with documentation only, with executive visibility, with committee review, with Kevin/full board approval, and what can never be done? Suggested default: agents may create issues, draft docs, draft mission packets, create low-risk templates, and comment on PRs/issues without approval; may create branches and edit docs on non-main branches with executive visibility; need committee/Kevin approval for code modifications, integrations, data handling, auth/security/privacy changes; and may never merge, deploy, change production, spend significant money, or handle secrets/regulated data without explicit approval. Confirm whether agents can create branches and draft PRs automatically or should ask first.

4. Memory and data rules: What may 007 remember or store? Suggested allowed memory: mission history, public repo notes, architecture decisions, lessons learned, reusable prompts, and non-sensitive project preferences. Restricted memory: private repo details, business-sensitive strategy, user/customer data, logs, uploaded files. Never store: secrets, passwords, API keys, regulated data, raw sensitive logs, or personal data unless explicitly approved. Confirm retention expectations and whether users should be able to delete/export memory later. Confirm whether 007 should stay file-based in GitHub for now or eventually use Supabase as a mission/memory database.

5. Governance and committee routing: Confirm the governance model. Suggested structure: low-risk work can proceed at executive-agent level with documentation; medium risk requires executive review and communication; high risk requires committee review before execution; critical risk requires Kevin plus full board-style review. Committees may include Executive Team, Risk Committee, Finance/Budget Committee, Technology/Architecture Committee, Product/User Impact Committee, and Full Board. Ask whether high-risk items always go to committee first or whether Kevin can override directly.

6. Board cadence: Define review rhythm. Suggested default: executive review per mission or weekly; committee review per high-risk mission; full board for major roadmap/project decisions or monthly review; emergency review whenever critical risk appears. Ask whether Kevin wants a weekly 007 review ritual/checklist.

7. Critical roadblocks: Define what blocks launch or execution. Suggested critical blockers: illegal activity, no rollback/backtrack plan for risky changes, unresolved security flaw, sensitive data exposure risk, unclear target user, failed QA with no mitigation, misleading public claim, unresolved auth/privacy issue, no owner for incident/failure, or no evidence for a critical assumption. Confirm whether a product can launch at 80% if medium risks remain documented but no high/critical blockers exist.

8. Definition of 80% complete: Define what 80% launch readiness means. Suggested definition: core user value works, target user is clear, no critical blockers, no illegal/deceptive behavior, no unresolved high security/data issue, known gaps are documented, rollback/backtrack path exists, and post-launch improvement list exists. Confirm who approves 80% readiness based on risk: executive team, Kevin, committee, or board.

9. Success metrics for 007: Define how Kevin will know 007 is working. Suggested metrics: fewer repeated mistakes, faster mission intake, clearer PRs, fewer abandoned experiments, useful learning artifacts, mission packets reused across projects, better risk identification, less context loss, better prioritization, and faster time from idea to usable output. Ask for the #1 metric proving 007 saves time or makes Kevin better.

10. Tool stack roles: Define the role of each tool. Suggested: ChatGPT handles orchestration, strategy, board sessions, writing, governance; Codex handles repo implementation, code edits, tests, PR work; Perplexity handles research, market/technical discovery, source-backed comparisons; GitHub is source of truth for issues, PRs, mission records; Supabase is later mission/memory DB only after file-based system is proven. Ask whether Perplexity should be used only when fresh research is needed or as part of every mission intake.

11. Human team roles: Define Kevin, Alfredo, and Josh responsibilities. Suggested: Kevin owns mission direction and final high-risk approval; Alfredo owns software engineering execution when assigned; Josh owns beta testing, strategy feedback, and user validation when assigned. Ask whether Alfredo/Josh should be included in mission packets by default only when their work is affected.

12. Target user / beneficiary rules: Every mission must identify primary user, secondary user, human benefit, and possible harm. Ask whether “humanity” should only be allowed as a broad mission principle while every actual product must name a specific target market.

13. Illegal / harmful work boundary: Confirm absolute no-go boundaries. Suggested: 007 must refuse or escalate illegal activity, data theft, unauthorized access, malware/abuse, deception/manipulation, privacy invasion, unsafe autonomy, exploitative product design, hidden data collection, and materially misleading public claims. Ask whether harmful but legal work should be rejected if it violates truth/data protection doctrine.

14. Communication rules: Define communication expectations. Suggested: low-risk work is summarized in the mission record; medium-risk work is summarized to executive agents/Kevin; high-risk work requires committee packet; critical-risk work requires full board packet before execution. Ask whether Kevin wants short executive summaries after every mission.

15. Scope boundary for Mission 001: Confirm that Mission 001 is in scope for mission packet template, governance routing standard, committee routing template, sample mission packet for agent-007, and QA/evidence expectations. Confirm that runtime automation, Supabase memory DB, autonomous code execution, deployment automation, and production integrations are out of scope for now.

Final output requirements: After Kevin answers, produce a clean operating standard called `007 Governance Threshold Standard`. It should be concise, practical, and enforceable. Include explicit rules for proceed / document / escalate / pause / reject. Include a “Challenger Review” section that identifies where the policy may be too strict, too loose, or likely to slow execution. Include a “Recommended Defaults” section for any area where Kevin does not provide a firm answer. Include a “Next Repo Changes” section listing which files should be created or updated in `agent-007`.
